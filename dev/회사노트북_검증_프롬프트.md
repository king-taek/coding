## 붙여넣을 프롬프트 (여기부터)

너는 반도체 AOI 검사 결과(NAS)에서 `Surface.flt` 결함 geometry 를 추출·검증하는 작업을 한다.
**실행할 파이썬은 하나뿐**이다: `dev/flt_check_auto_discover.py`. 자동탐색·파싱·매칭을 모두
하고 **결과를 단일 마크다운(.md) 한 개**로 떨군다(stdlib 만 사용 — 설치 불필요).
스키마는 이미 확정·검증되어 있다(실측 84건 + 실 NAS 116폴더 정합).

### 확정 Surface.flt 스키마 (TB500) — 참고용
레코드 152byte, 헤더 없음, little-endian. `파일크기 % 152 == 0`.
ActualX=24 d, ActualY=32 d, area=80 d, BlobBreadth=88 d, BlobFeretMax=104 d,
Contrast=136 d, zone=61 B, recipe=62 B.
환산: `area_um2=area×0.5929`, `width_um=BlobBreadth×0.77`, `length_um=BlobFeretMax×0.77`,
`contrast=Contrast(그대로)`. 매칭: INI `FaultX/FaultY`(없으면 X/Y) ↔ record `ActualX/ActualY`,
거리 ≤5µm 면 동일 결함.

### 할 일

**Step 0 — 그냥 한 줄 실행(무옵션 자동)**  ← 이거면 충분
```
git pull            # 먼저 repo 최신화(0.77 출처 추적 섹션 포함 버전)
python dev/flt_check_auto_discover.py
```
> 결과 md 가 **6개 섹션**만 있으면 옛 버전이다. 최신 버전은 **11개 섹션**(4. 픽셀크기 0.77 출처
> 후보 포함). 반드시 `git pull` 후 실행할 것.
인자 없이 실행하면: 대표 PI4 폴더 자동 포함(contrast 비0 확인용) + 14개 root 고르게 표집 +
**0.77 출처 추적**·필드 덤프·제품별 contrast·스키마 정합까지 전부 검사해 **`AOI_검증결과.md`**
한 개를 만든다. 그 md 의 **4. 픽셀크기(0.77) 출처 후보** 섹션에서 0.77 이 어느 파일/필드에서
나오는지 확인한다(앱이 그 값을 동적으로 읽도록 후속 작업할 근거).

> 아래 Step 1~4 는 특정 폴더만 따로 보거나 UI 교차확인을 더 정밀히 할 때만 쓰면 된다.

**Step 1 — 대표 폴더 1개로 재현 확인**
```
python dev/flt_check_auto_discover.py "P:\AOI-22\Scanresult\R_TB500_LIVE_PI4 AOI-21 Copy_0614\Setup1\PXU-PI4-ENGINEER-TEST\00RMF041XYC7" --out step1.md --max-dirs 1
```
이 폴더엔 검증된 예시 2건이 있다: `147206.243725.c.2104939970.2.jpeg`(zone 1, recipe 2, PI
Opening, contrast≈108.18) / `195998.277459.c.1570329500.2.jpeg`(zone 63, Scan Area,
contrast=0). `step1.md` 의 **5. 검증결과** 표에 두 건이 그대로 재현되는지 확인한다.

**Step 2 — 14개 root 를 고르게 빠른 표집** (recipe 다양성 확보가 핵심)
```
python dev/flt_check_auto_discover.py --max-dirs 100 --per-root-max 8 --latest-lots 3 --out AOI_검증결과.md
```
`--per-root-max` 로 한 root 에서 cap 이 소진되지 않게 해, **recipe 2(PI) 와 recipe 0(dummy/FS/
TOP-DIE) 가 둘 다 표집**되도록 한다. (이전 1회 실행은 첫 root 에서 100개가 소진돼 recipe 0 만
잡혔다 — 이번엔 그걸 피한다.)

**Step 2.5 — 0.77 출처 찾기: 소수 LOT 통째 덤프**
0.77 은 `AOI_검증결과.md` 의 4번 섹션에서 안 나왔다(어느 optic 도 0.46~0.47, 0.77 아님).
0.77 은 **정확히 '0.77' 로 안 적혀 있고 반올림 전 비슷한 값**(예: 0.7698)일 수 있다. 그래서
**PI 계열 LOT 폴더 2~3개만** 골라(예: 검증 예시가 있는 `00RMF041XYC7` 과 그 상위 Setup/제품
폴더) 안의 **모든 파일을 통째로** 덤프해 직접 뒤진다:
```
python dev/dump_lot_files.py "P:\...\PXU-PI4-ENGINEER-TEST\00RMF041XYC7" "P:\...\PXU-PI4-ENGINEER-TEST" --out LOT_파일덤프.md
```
> LOT 은 **적게**(2~3개). md 맨 위 **'0.77 근처 값 요약(0.74~0.80)'** 표에서 0.77 비슷한 값이
> 어느 파일/위치에 있는지 확인 → 그게 0.77 의 출처. 그 `LOT_파일덤프.md` 를 그대로 전달.

**Step 3 — zone×recipe 교차로 contrast 규칙 확인**
`AOI_검증결과.md` 의 **2. zone × recipe 별 contrast 분포** 표를 본다. 확인 질문:
- contrast=0 이 **zone 때문인지 recipe/제품 때문인지**? (예: recipe 2 의 zone 1 은 비0,
  recipe 0 은 zone 무관 0 … 같은 패턴인지.)

**Step 4 — AOI UI 수기 교차확인 (사람이 수행, 이게 최종 결론 근거)**
`AOI_검증결과.md` 의 **3. UI 수기확인 shortlist** 표에는 recipe2-zone1 / 기타recipe-zone1 /
zone63 버킷에서 자동 선별된 결함들이 있고 `UI_*` 칸이 **비어 있다**. 각 결함을 AOI UI 에서 열어
**UI 의 area/width/length/contrast/zone명** 을 그 빈칸에 직접 적는다. 핵심 확인:
- (a) area/width/length 가 UI 와 ±정수 수준 일치하는가?
- (b) **추출 contrast=0 인 결함을 UI 도 0/공란으로 보여주는가?** (그렇다면 0 은 정상.
  UI 가 0 이 아닌 값을 보이면 contrast 소스가 Surface.flt 가 아니므로 추가 조사 필요.)
- (c) zone/recipe 코드가 UI 의 zone 이름·recipe 와 대응되는가?

**Step 5 — 전달**
`AOI_검증결과.md` 한 파일에 요약·스키마 정합·zone×recipe contrast·매칭 실패 분류·UI 수기확인
표가 모두 들어있다. UI 칸을 채운 그 **.md 파일 하나를 그대로 복사해** 사용자에게 전달한다.

> 옵션: 전체 야간 정밀은 `--deep --max-dirs 1000 --per-root-max 80 --timeout 7200`.
> `Surface.flt` 없는 폴더(KLA 등)는 자동으로 "측정정보 없음" 처리된다.

## 무엇을 알고 싶은가 (요약)

- area/width/length 는 이미 실측(84건 + 실 NAS 116폴더)으로 검증 완료. 이번 핵심은 **contrast**.
- 갱신된 가설: **contrast=0 은 zone 만이 아니라 recipe/제품 의존**이다. 1회 NAS 실행에서 표집된
  폴더가 전부 recipe 0(2D@RE/FS/TOP-DIE)이라 zone 1 까지 contrast 100% 0 이었다. 반면 검증
  예시(recipe 2, PI)의 zone 1 은 contrast 108.18. → recipe 별로 contrast 측정 여부가 갈리는 것으로 보인다.
- **UI 수기확인(Step 4)** 으로 "추출 contrast=0 ↔ UI 도 0/공란" 이 맞으면 0 은 정상.
  그 결과에 따라 엑셀 표시를 **contrast=0 일 때 줄 생략 또는 '—'** 로 바꿀지 결정한다(현재는 `contrast 0.00`).
- 직전 1회 실행에서 `UI_*` 가 자동복사돼 실제 UI 확인이 안 됐었다 → 이번 스크립트는 `UI_*` 를
  **빈칸**으로 두어 반드시 사람이 채우게 했다.
