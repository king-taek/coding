# 매칭 레시피 원리 — TOP5 + 운영 엔진 매핑

> 매칭(Stage 2) 속도 개선 실험의 **최종 결론** 문서입니다. 무엇을 왜 채택했는지,
> 각 레시피가 어떤 원리로 빠른지, 운영 두 모드에 어떻게 반영했는지 정리합니다.
> (실험 과정은 `docs/진행상황_매칭속도개선.md`, 벤치 사용법은 `docs/개발자_벤치마크_가이드.md`.)

## 공통 뼈대 — 2단계 매칭

검증 사진 1장을 기준 수백 장과 비교할 때 전부 정밀 비교하면 느리다. 그래서:

1. **1단계(거르기·recall)**: CNN 임베딩(GPU 또는 CPU)으로 벡터를 뽑아 **코사인 상위 후보**만
   추린다 — 싸고 빠름.
2. **2단계(재채점·rerank)**: 그 상위 후보만 **고전 픽셀 비교**로 정밀 채점하고, 두 점수를
   z-융합해 최종 순위.

병목은 항상 **2단계**(전체의 ~80%). 고전 비교는 세 항으로 구성된다:

| 항 | 의미 | 비용 | 정확도 기여 |
|---|---|---|---|
| **pHash** | 이미지 전체의 거친 지각 해시 | 가장 쌈 | 보조 |
| **ORB** | 회전·조명에 강한 키포인트 정합 | 중간 | **핵심** |
| **SSIM** | 전 픽셀 구조적 유사도 | **가장 비쌈** | 보조 |

실측 결론: **ORB 만 유지하면 정확도(recall@1 97.6%)가 보존**되고, ORB 를 빼면 41~71% 로
붕괴한다. 그래서 TOP5 는 **전부 ORB 를 유지**하고, "무엇을 빼거나 어떻게 싸게 할까"만 다르다.

## TOP5 레시피 — 각각의 원리

실측에서 정확도 97.6%(현행 동률)를 보존하면서 빠르고 안정적인 5가지.

### 1. `cpu_rr_phash_orb` — 최고비용 SSIM 제거
2단계에서 `{pHash + ORB}` 만 쓰고 **SSIM(전 픽셀 비교)을 제거**. pHash 가 전역 구조를, ORB 가
국소 특징을 보므로 SSIM 정보가 상당히 중복된다. 비싼 항 하나를 통째로 빼서 **최속급**, 정확도 보존.

### 2. `rr_parallel` — 항은 그대로, 코어를 다 쓴다 (가장 안전)
2단계 항을 **현행과 100% 동일**(pHash+ORB+SSIM)하게 두되 기준들을 **멀티코어 병렬** 채점.
계산이 같아 **결과(점수)가 현행과 동일**, 시간만 1/N. 정확도를 바꿀 여지가 원리적으로 0이라
**운영 승격 위험이 가장 낮다**. (운영 채점기는 이미 이 병렬화를 한다.)

### 3. `cpu_rr_parallel16` — 병렬 + 임베딩도 CPU
`rr_parallel` 과 같되 1단계 임베딩까지 CPU 로(`recall=cpu`). **가속기 없는 PC** 의 병렬
풀-고전 성능 기준선. "임베딩 장치를 바꿔도 총시간은 병목(2단계) 때문에 거의 그대로"를 재확인.

### 4. `cpu_rr_orb_only` — ORB 하나로 충분한가
2단계를 **ORB 단독**으로(pHash·SSIM 제거). ORB 만으로도 97.6% 유지 → **이 도메인의 변별력은
사실상 ORB 가 만든다**는 증거. pHash/SSIM 은 보험일 뿐. 비용도 ORB 하나라 빠름.

### 5. `rr_orb_center50` — ORB 를 defect 중심으로 가중 (★ 고효율 모드 채택)
ORB 단독 재채점에 **defect 위치 사전지식**을 더한다. AOI 이미지는 **defect 이 항상 정중앙**
(장비가 defect 좌표로 FOV 크롭)이라:
- ORB 키포인트 매치를 **중앙으로부터의 거리로 가중**(중앙=1.0, 가장자리는 1−0.5×거리).
- 배경의 반복 RDL 패턴(트레이스·via)은 사진 전체에 깔려 **우연히 잘 매칭되는 혼동원**.
  중앙 가중이 이 배경 매치 영향을 깎고 **정중앙 defect 근처 매치를 부각** → defect 판별력↑.
- **단일 패스**: 별도 크롭/2차 채점 없이 ORB 점수 집계 때 가중치만 곱하므로 추가 비용 ~0.

실측에서 TOP5 중 **최속(현행 대비 ×5~6)** 이면서 정확도 97.6% 유지.

## 운영 엔진 매핑 (현재 적용)

| 모드 | 채택 레시피 | 1단계(recall) | 2단계(rerank) | 장치 |
|---|---|---|---|---|
| **기본 모드** | `rr_parallel` | (전수 — 추림 없음) | pHash+ORB+SSIM 전체·CPU 멀티코어 병렬 | **CPU 만** |
| **고효율 모드** | `rr_orb_center50` | GPU(MobileNetV3) 임베딩으로 상위 후보 추림 | **ORB 단독 + 중앙(defect) 가중 0.5**·CPU 병렬 | **GPU + CPU** |

- 코드: `main_window._make_sim_cfg` 가 모드별로 `rerank_components`/`orb_center_weight` 를
  설정 → 기본 모드 `SlotPrecomputeWorker`(전수 병렬), 고효율 `EfficiencyScheduler`(추림+재채점).

## GPU / NPU 사용 여부

- **NPU**: **두 모드 모두 사용하지 않는다.** 실측상 NPU 임베딩은 속도 이득 0이고 배치 추론에서
  정확도가 깨졌으며, 가동률을 강제로 올린 실험(`npu-hi`)도 이득 0 + 타임아웃/폴백으로 불안정했다.
  코드는 보존하되 운영 경로에서 비활성(`_select_backend` 가 NPU 미선택).
- **GPU**: **고효율 모드의 1단계 임베딩**에서 사용(MobileNetV3, batch16). GPU 가 없으면
  자동으로 CPU 고전 단독으로 폴백한다. **기본 모드는 GPU 도 쓰지 않는 순수 CPU** 경로.

요약: 이번 변경 후에도 **GPU 는 고효율 모드에서 계속 쓰이고, NPU 만 완전히 사용하지 않는다.**

## 개발자 모드(벤치마크)

추가 실험은 끝났고, 옵션은 **TOP5 비교만** 남겼다(`--recipes top5`). 나머지 레시피·그룹(NPU
스윕/단독/고가동·model-zoo·center 등)은 **나중에 다시 볼 때를 위해 보존**하되 기본 옵션에서는
내렸다. 정확도를 더 짜내려면 GT(정답 라벨) 확대가 선행돼야 한다(현재 41쿼리·전부 97.6% 동률).
