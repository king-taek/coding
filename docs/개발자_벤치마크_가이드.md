# 개발자 벤치마크 가이드 — 매칭 가속 조합(CPU·GPU·NPU) 실험·기록

매칭 단계(Stage 2)의 속도를 **정확도 손실 없이** 끌어올리기 위해, CPU·Intel
GPU·Intel NPU 의 조합(레시피)을 한 번에 실험·기록·비교하는 개발자 전용 도구다.

- 핵심 원칙
  1. **정확도 우선** — '실제 매칭이 맞았는지'(유사도 점수가 아니라)를 측정하고,
     정확도가 떨어지는 조합은 **절대 추천하지 않는다.**
  2. **캐시 우회** — 모든 조합을 `bench_no_cache=True` 로 실행해 **처음 매칭하는
     것처럼** 측정한다(특징/임베딩/점수 디스크 캐시 미사용 → 공정 비교).
  3. **로딩 안전** — 조합은 항상 **순차** 실행, 동시성은 가용 메모리로 상한,
     조합별 타임아웃, 대용량은 서브샘플 → 시스템이 로딩으로 꺼지지 않는다.

## 1. 켜는 법 (개발자 모드)

일반 사용자 화면에는 보이지 않는다. 다음 중 하나로 활성화:

- **앱 안에서 (가장 간단)**: 셋업(시작) 화면에서 **`Ctrl + Shift + D`** → 개발자 모드
  on/off. 켜면 즉시 하단 버튼 바에 **[개발자 벤치마크]** · **[정답 라벨 만들기]**
  버튼이 나타나고, 상태는 `ui_prefs.json` 에 저장돼 다음 실행에도 유지된다.
- **환경변수**: 실행 전에 `AOI_DEV_MODE=1` (강제 on — 배포본/CI 용)
- **설정 파일**: `ui_prefs.json` 의 `dev_mode: true`

> 버튼이 안 보이면 환경변수가 없는 더블클릭 실행이라 그렇다 — 그 경우
> **`Ctrl + Shift + D`** 로 켜면 된다(환경변수 없이 앱 안에서 토글).

## 2. 두 가지 실행 방법

### 2.1 GUI

셋업 화면 → **[개발자 벤치마크]** →

1. **기준 폴더** 지정(검증 폴더가 없으면 **자기검증** 체크 — 기준 사진을
   약하게 증강해 정답이 있는 검증셋을 자동 생성. 검증 장비 데이터가 없어도
   실제 정확도를 측정할 수 있다).
2. **프리셋**으로 실험할 **조합 선택** — 기본은 **빠른(린)**:
   - **빠른(린)**: 정확도 기준선·현행 + 재채점 **생존자**(`rr_parallel`/`cpu_rr_orb_only`/
     `cpu_rr_parallel16`) + **중앙-인식**(defect 정중앙 활용) 신규 실험. 실측상 임베딩 장치
     교체는 ×1.02 라 빼고, 3배의 열쇠인 'CPU 재채점 줄이기'와 사용자 제안만 본다.
     **현행을 항상 포함**해 추천이 항상 유효(예전엔 현행 미포함 런에서 추천이 잘못 나왔다).
   - **대결**: 현행 vs 재채점 생존자 직접 비교(반복 측정으로 분산 확인).
   - **표준**: 핵심 레지스트리.  **전체(아카이브)**: 입증된 사패(NPU 스윕·단독·ORB제거·모델주머니)까지 opt-in.
   - 개별 체크박스·그룹 토글(중앙-인식 포함)로 세밀하게 고를 수도 있다.
3. 서브샘플(slot/이미지 상한), 조합별 타임아웃 설정.
4. **[실험 시작]** → 결과 표(총시간/임베딩/재채점/img·s/피크MB/**정확도**/비고)
   가 채워지고, **추천 조합**에 ⭐ 와 현행 대비 배속이 표시된다.
   - 실행은 **별도 프로세스로 격리**된다. 한 조합이 NPU/GPU 드라이버 문제로 죽어도
     **프로그램은 안 꺼지고**, 어느 조합이 죽였는지 비고에 남으며 나머지는 이어서 측정된다.

### 2.2 CLI (헤드리스 — 자동 기록)

```bash
# 레시피 목록 보기
python -m aoi_verification.app.dev.benchmark --list

# 빠른(린) 프리셋(기본·권장) — 현행·기준선 + 재채점 생존자 + 중앙-인식 신규 실험
python -m aoi_verification.app.dev.benchmark --ref "기준" --self-test --recipes quick --timeout 120

# 대결: 현행 vs 재채점 생존자 직접 비교(3회 반복으로 분산까지 확인 권장)
python -m aoi_verification.app.dev.benchmark --ref "기준" --self-test --recipes faceoff --timeout 120

# 최종 벤치: 고전 워밍업→정식(2회) + 현행 + TOP5 + NPU 고가동 5 (후보 recall·NPU 사용량 표 포함)
python -m aoi_verification.app.dev.benchmark --ref "기준" --self-test --recipes final --timeout 180

# NPU 사용량을 높인 5가지(동시추론/스트림/분담)만 따로
python -m aoi_verification.app.dev.benchmark --ref "기준" --self-test --recipes npu-hi --timeout 180

# 중앙-인식: defect 이 정중앙인 특성 활용(영역융합 A·캐스케이드 B)
python -m aoi_verification.app.dev.benchmark --ref "기준" --self-test --recipes center --timeout 120

# NPU 배치 정확도 붕괴 원인 규명 — GPU vs NPU(b1) vs NPU(b16) 임베딩 일치도
python -m aoi_verification.app.dev.benchmark --ref "기준" --npu-embed-diag

# 실제 두 장비 데이터로(정답 라벨 있으면 recall@1, 없으면 기준선 일치율)
python -m aoi_verification.app.dev.benchmark --ref "<기준폴더>" --val "<검증폴더>" \
    --labels labels.json --max-slots 3 --timeout 120

# 아카이브(opt-in) — 입증된 사패까지 전부(NPU 스윕/단독/고속재채점/모델주머니)
python -m aoi_verification.app.dev.benchmark --ref "기준" --self-test --recipes all+   # 확장 포함 전부
```

> ``--recipes`` 는 ``quick``(린 기본) · ``faceoff``(현행vs생존자) · ``center``(중앙-인식) ·
> ``core``(핵심 레지스트리) · ``all+``(아카이브 포함 전부) · 그룹명(``center`` / ``npu-sweep`` /
> ``npu-only`` / ``fast-rerank`` / ``model-zoo``) · 개별 키를 콤마로 섞어 쓸 수 있다.
> GUI 는 프리셋 버튼·그룹 체크박스로 같은 선택을 제공한다.
>
> GUI 의 프로세스 격리는 같은 CLI 를 자식으로 띄워 쓰며, 이때 ``--emit-progress``(진행·런디렉토리를
> 기계가 읽는 줄로 출력)와 ``--explicit``(스킵 면제 키 지정)를 사용한다. 보통은 신경 쓰지 않아도 된다.

기록은 `~/.aoi_verification_cache/dev_bench/<호스트>/<타임스탬프>/` 에
`result.json` 과 `report.md` 로 남는다(실행마다 누적).

## 2.5 정답 라벨 만드는 단계 (실제 정확도용)

'실제 정확도'(유사도 점수가 아니라 매칭 정답 기준)를 재려면 기준 사진마다
**정답인 검증 사진**을 지정한 라벨이 필요하다. **정답은 여러 개일 수도, 없을
수도 있다.**

라벨 JSON 형식 — 값은 항상 리스트:
```json
{ "<slot>": {
    "<기준 사진 경로>": ["<정답 검증 사진>", "<또 다른 정답>"],   // 복수 정답
    "<다른 기준 사진 경로>": []                                  // 정답 없음
} }
```
- 값이 **2개 이상** = 복수 정답(상위 K 안에 그중 하나라도 있으면 정답).
- 값이 **빈 리스트 `[]`** = '정답 없음'으로 검토 완료.
- 라벨에 **없는** 기준 사진 = 미검토 → 정확도 집계에서 제외.

### (a) GUI — 클릭으로 만들기
셋업 화면 → **[정답 라벨 만들기]** → 기준/검증 폴더 지정 → 기준 사진마다
후보 검증 사진을 클릭해 정답 표시(여러 개 선택 가능, **[정답 없음]** 버튼).
후보는 파일명순(기본) 또는 **[유사도순 정렬]** 로 보여줘 빠르게 고를 수 있다
(표시 순서일 뿐 정답에는 영향 없음). **[저장]** 하면 라벨 JSON 이 생기고,
[개발자 벤치마크]의 *정답 라벨 파일* 칸에 자동으로 채워진다.

### (b) CLI — 템플릿 만들고 채우기 / 통계
```bash
# 빈 정답 템플릿 생성(모든 기준 사진을 빈 리스트로 나열) → 직접 채운다
python -m aoi_verification.app.dev.benchmark --ref "<기준>" --val "<검증>" \
    --make-labels-template labels.json

# 채운 라벨의 통계(정답있음/없음/복수) 확인
python -m aoi_verification.app.dev.benchmark --labels-stats labels.json
```

## 3. 레시피 — 각 연산을 어느 장치에서 어떻게

한 레시피는 매칭을 두 연산으로 나눈다.
- **후보 선별(recall)**: 이미지에서 임베딩을 *뽑아내* 코사인으로 상위 후보 추림.
- **정밀 재채점/계산(scoring)**: pHash+ORB+SSIM 고전 점수 + z-융합. **항상 CPU.**

따라서 *"NPU 로 데이터를 뽑고 CPU 로 계산"* = `recall=NPU, scoring=fusion`
(아래 `npu_extract_cpu_fuse`).

| 키 | recall(임베딩) | scoring(계산) | 핵심 의도 |
|---|---|---|---|
| `cpu_classical_full` | 없음 | CPU 고전 전수 | **정확도 기준선(GOLD)**, 가장 느림 |
| `gpu_fusion_b16` | GPU MobileNetV3 b16 | CPU 융합(top40) | **현행 운영(속도 기준)** |
| `npu_extract_cpu_fuse` | **NPU** ResNet18 | CPU 융합 | **NPU 추출 + CPU 계산(사용자 제안)** |
| `npu_mbnet_cpu_fuse` | NPU MobileNetV3 | CPU 융합 | GPU 와 동일 백본을 NPU 에서(분담/대조) |
| `cpu_embed_fusion` | CPU MobileNetV3 | CPU 융합 | 가속기 없는 PC 폴백 성능 |
| `gpu_embed_only` | GPU MobileNetV3 | 없음(코사인만) | 최속·정확도↓(융합 필요성 입증) |
| `gpu_npu_split_fusion` | **GPU+NPU 분담** | CPU 융합 | **임베딩을 두 장치에 절반씩** → 처리량↑ |
| `gpu_fusion_topk20` | GPU MobileNetV3 b16 | CPU 융합(top20) | 재채점 얕게 → 빠름(정확도 검증 필수) |
| `gpu_fusion_topk60` | GPU MobileNetV3 b16 | CPU 융합(top60) | 재채점 깊게 → 정확도 여유 |
| `gpu_fusion_crop` | GPU MobileNetV3 b16 | CPU 융합+중앙30% | 교차 호기 정확도↑ |
| `gpu_fusion_b1` | GPU b1 | CPU 융합 | **함정 재현**(처리량 폭락) |
| `gpu_fusion_b4` | GPU b4 | CPU 융합 | 처리량 정상화 지점 |
| `gpu_npu_ensemble_fusion` | GPU+NPU **앙상블(중복)** | CPU 융합 | **안티패턴 대조**(시간 2배·이득 0) |

> 정확도는 임베딩 백본 종류가 아니라 **CPU 고전 융합**이 좌우한다
> (docs/NPU 효율성 분석 보고서). 그래서 장치 조합은 주로 *"임베딩을 누가/어떻게
> 더 빨리 뽑느냐"* 의 속도 문제이며, 융합이 정확도를 지킨다.

## 3.5 확장 실험 그룹 (NPU 집중 + 새 모델)

실측에서 NPU 추출이 정확도(97.6%)를 지키며 가장 빨랐고, **CPU 재채점이 병목**
(임베딩 ~3.7s vs 재채점 ~57s)임이 드러났다. 이를 더 파고들기 위한 4개 확장
그룹을 추가했다(``--recipes <그룹명>`` 또는 GUI 그룹 체크박스).

### (A) `npu-sweep` — NPU 사용 방식 스윕 (≥20가지)
같은 추천 구성 주변에서 **한 축만** 바꿔 원인을 분리한다(전부 NPU 추출 + CPU 융합):
- **모델**: `npu_model_*` (mobilenet_v3_small / resnet18 / mobilenet_v2 / squeezenet1_1 / efficientnet_b0)
- **정적 배치 B**: `npu_b{1,4,8,16,32}`
- **병렬 수준(동시 추론 수)**: `npu_c{1,2,4,8,16,32,64,96}` — AsyncInferQueue in-flight
- **성능 힌트**: `npu_hint_{throughput,latency,cumulative_throughput}`
- **스트림(다중 동시 작업)**: `npu_streams{1,2,4}` — NUM_STREAMS
- **전처리 멀티스레드**: `npu_prep{2,4,8}` — NPU 가 굶지 않게 텐서 공급
- **입력 해상도**: `npu_px{224,256}`

### (B) `npu-only` — NPU 단독 채점 (CPU 불필요 검증)
CPU 재채점 없이 **NPU 임베딩 코사인만**으로 매칭(`scoring=embed_only`).
NPU 단독이 충분히 빠르고 정확하면 CPU 가 필요 없음을 직접 확인한다.
`npu_only_{mobilenet_v3_small,resnet18,mobilevit_s,efficientnet_b0}` 와, 거의
단독에 가까운 `npu_only_mbnet_phash`(상위 10개만 가장 싼 pHash 1항목 보정) 포함.

### (C) `fast-rerank` — CPU 재채점 고속화 (병목 직접 공략)
실측상 매치 시간의 대부분은 **CPU 재채점**(임베딩 ~3.7s vs 재채점 ~57s)이고, 그중
**ORB(디스크립터 정합)·SSIM 이 비싸다**(pHash 는 사전계산 해시라 매우 쌈). 이 그룹은
재채점을 빠르게 만드는 방법들을 한 축씩 바꿔 측정한다(정확도 검증이 전제).

기존(가속 추출 + CPU 재채점):
- `rr_phash` / `rr_phash_ssim` / `rr_orb_ssim` — 항 빼기(가장 비싼 ORB 제거 등)
- `rr_parallel` — 항목 그대로(정확도 동일) 16스레드 병렬 · `rr_phash_topk20` — 깊이까지 축소
- `rr_npu_phash_parallel` — NPU 추출 + (ORB 뺀) pHash+SSIM 16스레드 병렬 결합

**CPU 끝까지(`cpu_rr_*`) — CPU 매치 단계 고속화 ≥10가지**(recall=CPU 임베딩 후보 +
CPU 재채점). 다섯 레버를 한 축씩, 또는 결합해 측정:
1. **항 빼기**: `cpu_rr_phash`(pHash 단독) · `cpu_rr_phash_ssim`(ORB 제거) ·
   `cpu_rr_phash_orb`(SSIM 제거) · `cpu_rr_orb_only` · `cpu_rr_ssim_only`(단독 대조)
2. **병렬화**: `cpu_rr_parallel8` / `cpu_rr_parallel16` / `cpu_rr_parallel32`
   (전체 항·정확도 동일, 워커 수만 변경 — 과도구독 한계점 확인)
3. **ORB 특징 수 줄이기**: `cpu_rr_orb256` / `cpu_rr_orb128`
   (ORB 검출/정합 비용은 특징 수에 비례 — 기본 500 → 256/128)
4. **재채점 깊이 줄이기**: `cpu_rr_topk10` / `cpu_rr_topk20`(상위 K 만 정밀 채점)
5. **중앙 crop**: `cpu_rr_crop`(비교 면적↓ → ORB·SSIM↓, 교차호기 정확도에도 도움 가능)
- 결합 후보: `cpu_rr_light_parallel`(ORB 제거+병렬+topk20, 실용 최속) ·
  `cpu_rr_orb256_parallel`(ORB256+병렬, 정확도 보존형)

> 자기검증(기준 폴더)에서 측정: ORB 제거/병렬/특징수 축소가 `cpu_classical_full` 대비
> 크게 빠르면서 정확도를 유지하는지 확인(데이터에 따라 다름 — 반드시 실제 데이터로 재확인).
> 새 CPU 노브: ORB 특징 수(`SimilarityConfig.orb_nfeatures`)·재채점 컴포넌트
> (`benchmark._rerank_components`: phash/phash_ssim/orb_ssim/phash_orb/orb/ssim).

### (D) `model-zoo` — 새 모델/방식
대상 장비에 패키지가 있으면 NPU 추출 + 전용 채점기로, 없으면 CPU 폴백 + 필요한
패키지를 비고에 안내한다(`run_this_before.py` 가 설치).

| 모델/방식 | 계열 | 필요 패키지 | 상태 |
|---|---|---|---|
| MobileViT-S / XS | ViT 임베딩 | `timm` | **대상 장비에서 실동작**(NPU 추출+융합) |
| CAE · U-Net · Attention U-Net | 오토인코더/세그 인코더 병목 | torch(가중치 필요) | 구조 실동작(임의 가중치 — 정확도엔 학습 필요) |
| SuperPoint + LightGlue | 키포인트 정합 | `kornia` | 스캐폴드(현재 CPU 폴백 — 검증 후 활성) |
| PatchCore · PaDiM | 이상탐지 패치 거리 | `anomalib` | 스캐폴드(현재 CPU 폴백 — 검증 후 활성) |

> 임베딩 계열(MobileViT/CAE/U-Net)은 `model_zoo.build_backbone` 으로 OpenVINO
> 컴파일되어 NPU/GPU 에서 실제 추출된다.  키포인트/이상탐지 계열은 레지스트리·
> 가용성 탐지·폴백·설치까지 갖춘 **스캐폴드**이며, 전용 채점기는 대상 장비에서
> 패키지 검증 후 켠다(그 전엔 안전하게 CPU 고전으로 폴백 + 비고 안내).

## 4. 정확도 측정 방법

- **정답 라벨이 있으면**: `recall@1`, `recall@5`(상위 K 안에 정답이 있으면 정답).
- **라벨이 없으면**: `cpu_classical_full`(고전 전수, 정답 기준선) 의 top-1 과의
  **일치율** — '현행 정확도를 보존하는가'를 측정.
- **추천 규칙**: 완료된 조합 중 **현행(`gpu_fusion_b16`) 대비 정확도가 낮지
  않으면서 가장 빠른** 것. 정확도가 떨어지면 더 빨라도 추천하지 않는다.

---

## 5. 3배 속도 — 권장 탐색 순서

현행(`gpu_fusion_b16`)을 분모로, 정확도를 지키며 시간을 줄이는 경로:

1. **배치 고정** — GPU 는 `batch=16`(또는 4~16). `batch=1` 은 처리량 폭락(함정).
2. **재채점 깊이 조정** — `topk20` 이 정확도를 유지하면 CPU 단계가 짧아져 큰 이득.
   유지 못 하면 `topk40` 으로 되돌린다.
3. **임베딩 분담** — `gpu_npu_split_fusion` 으로 GPU+NPU 가 임베딩을 절반씩 동시에
   뽑아 추출 처리량을 올린다(중복 앙상블 아님).
4. **중앙 crop** — 교차 호기에서 정확도 여유가 필요하면 `gpu_fusion_crop`.

가장 빠른 조합이 **정확도 기준선과 동률**일 때만 채택한다.

> **검증 메모**: 임베딩 가속(GPU/NPU)의 실제 배속은 **Intel GPU/NPU 가 있는
> 대상 장비에서** 측정된다. 가속기가 없는 환경에서는 임베딩 레시피가 자동으로
> **CPU 고전 폴백**(비고에 표시)되어, 하니스 동작/정확도/기록은 그대로 검증되되
> 장치 배속만 0 으로 보고된다. 하니스는 장치-비의존이라 대상 장비에서 그대로
> 실측하면 된다.

## 5.5 불필요한 실험 자동 스킵 (속도/시간 절약)

"필요없는 테스트는 하지 않는다" — 기본 스위트는 다음을 **측정하지 않고** 사유와
함께 '건너뛴 레시피' 표로 남긴다. 정확도 기준선(`cpu_classical_full`)은 항상 측정.

1. **함정/대조용**(`diagnostic`) — `gpu_fusion_b1`(batch1 함정)·`gpu_embed_only`·
   `gpu_npu_ensemble_fusion`. 운영 판단에 새 정보가 없어 제외.
2. **폴백 중복** — 필요한 GPU/NPU가 없거나 모델 패키지(timm/kornia/anomalib)가
   없어 어차피 CPU 고전으로 폴백 → 같은 결과 반복이라 스킵.
3. **과거 저성능** — 이전 기록(`dev_bench`·`bench결과`)에서 **운영 대비 정확도가
   3%p 넘게 낮았던** 레시피. 근소차(≤3%p)는 노이즈로 관용하고, 가속기 없는 CPU
   폴백 기록은 변별력이 없어 판정에서 제외한다.

> **개별 키로 직접 고른 레시피는 어떤 규칙으로도 스킵하지 않는다**(존중).
> 그룹(`npu-sweep` 등)·전체(`all+`)로 펼쳐진 것만 위 규칙이 적용된다.
> 전부 강제로 측정하려면 CLI `--all-recipes` 또는 GUI '모든 레시피 측정' 체크.

```bash
# 예: 가속기 없는 PC 에서 all+ 를 돌려도 폴백 중복은 자동 스킵돼 빠르게 끝난다
python -m aoi_verification.app.dev.benchmark --ref 기준 --self-test --recipes all+
# 스킵 없이 전부 측정(함정·저성능 포함)
python -m aoi_verification.app.dev.benchmark --ref 기준 --self-test --recipes all+ --all-recipes
```

## 6. 로딩 안전장치

- **프로세스 격리**: 레시피 실행을 **자식 프로세스**로 돌린다. OpenVINO/NPU 네이티브
  크래시(segfault)·멈춤은 파이썬 예외/타임아웃으로 못 막지만, 자식이 죽어도 **GUI 는
  안 꺼지고** 어느 레시피가 죽였는지 기록한 뒤 **나머지를 이어서 측정**한다. 레시피마다
  부분 결과(`result.json`)를 저장해 도중에 죽어도 거기까지는 복구된다
  (`drive_isolated_suite` · `--emit-progress`).
- 레시피는 **순차** 실행(두 장치-부하 조합이 동시에 돌지 않음).
- 동시 추론 수는 **가용 메모리의 약 50% 안**에서 자동 상한(`safe_concurrency`).
- 조합별 **타임아웃**(초). 초과 시 그 조합만 중단하고 다음으로. 부모는 자식이 너무 오래
  무응답이면 **강제 종료**하는 안전망도 둔다(타임아웃의 3배+여유, 무제한이면 30분).
- **서브샘플**(slot/이미지 상한)로 대용량을 안전하게 표본 측정.
- 실행 중 **피크 메모리(MB)** 를 기록해 위험 조합을 사후 식별.
