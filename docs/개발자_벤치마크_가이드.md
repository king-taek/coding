# 개발자 벤치마크 가이드 — 매칭 가속 조합(CPU·GPU·NPU) 실험·기록

매칭 단계(Stage 2)의 속도를 **정확도 손실 없이** 끌어올리기 위해, CPU·Intel
GPU·Intel NPU 의 조합(레시피)을 한 번에 실험·기록·비교하는 개발자 전용 도구다.

- 핵심 원칙
  1. **정확도 우선** — '실제 매칭이 맞았는지'(유사도 점수가 아니라)를 측정하고,
     정확도가 떨어지는 조합은 **절대 추천하지 않는다.**
  2. **캐시 우회** — 모든 조합을 `bench_no_cache=True` 로 실행해 **처음 매칭하는
     것처럼** 측정한다(특징/임베딩/점수 디스크 캐시 미사용 → 공정 비교).
  3. **로딩 안전** — 조합은 항상 **순차** 실행, 동시성은 가용 메모리로 상한,
     조합별 타임아웃, 대용량은 서브샘플 → 시스템이 로딩으로 꺼지지 않는다.

---

## 1. 켜는 법 (개발자 모드)

일반 사용자 화면에는 보이지 않는다. 다음 중 하나로 활성화:

- **환경변수**: 실행 전에 `AOI_DEV_MODE=1`
- **설정 파일**: `ui_prefs.json` 의 `dev_mode: true`

활성화되면 셋업 화면 하단 버튼 바에 **[개발자 벤치마크]** 버튼이 나타난다.

---

## 2. 두 가지 실행 방법

### 2.1 GUI

셋업 화면 → **[개발자 벤치마크]** →

1. **기준 폴더** 지정(검증 폴더가 없으면 **자기검증** 체크 — 기준 사진을
   약하게 증강해 정답이 있는 검증셋을 자동 생성. 검증 장비 데이터가 없어도
   실제 정확도를 측정할 수 있다).
2. 실험할 **조합 선택**(미선택 시 전체).
3. 서브샘플(slot/이미지 상한), 조합별 타임아웃 설정.
4. **[실험 시작]** → 결과 표(총시간/임베딩/재채점/img·s/피크MB/**정확도**/비고)
   가 채워지고, **추천 조합**에 ⭐ 와 현행 대비 배속이 표시된다.

### 2.2 CLI (헤드리스 — 자동 기록)

```bash
# 레시피 목록 보기
python -m aoi_verification.app.dev.benchmark --list

# 자기검증(2호기 불필요): 기준 폴더 사진을 증강해 정답 생성 후 전체 조합 측정
python -m aoi_verification.app.dev.benchmark --ref "기준" --self-test --recipes all --timeout 120

# 실제 두 장비 데이터로(정답 라벨 있으면 recall@1, 없으면 기준선 일치율)
python -m aoi_verification.app.dev.benchmark --ref "<기준폴더>" --val "<검증폴더>" \
    --labels labels.json --max-slots 3 --timeout 120
```

기록은 `~/.aoi_verification_cache/dev_bench/<호스트>/<타임스탬프>/` 에
`result.json` 과 `report.md` 로 남는다(실행마다 누적).

정답 라벨 JSON 형식:
```json
{ "<slot>": { "<기준 사진 경로>": ["<정답 검증 사진 경로>", "..."] } }
```

## 3. 레시피 — 각 연산을 어느 장치에서 어떻게

한 레시피는 매칭을 두 연산으로 나눈다.
- **후보 선별(recall)**: 이미지에서 임베딩을 *뽑아내* 코사인으로 상위 후보 추림.
- **정밀 재채점/계산(scoring)**: pHash+ORB+SSIM 고전 점수 + z-융합. **항상 CPU.**

따라서 *"NPU 로 데이터를 뽑고 CPU 로 계산"* = `recall=NPU, scoring=fusion`
(아래 `npu_extract_cpu_fuse`).

| 키 | recall(임베딩) | scoring(계산) | 핵심 의도 |
|---|---|---|---|
| `cpu_classical_full` | 없음 | CPU 고전 전수 | **정확도 기준선(GOLD)**, 가장 느림 |
| `gpu_fusion_b16` | GPU MobileNetV3 b16 | CPU 융합(top40) | **현행 운영(속도 기준)** |
| `npu_extract_cpu_fuse` | **NPU** ResNet18 | CPU 융합 | **NPU 추출 + CPU 계산(사용자 제안)** |
| `npu_mbnet_cpu_fuse` | NPU MobileNetV3 | CPU 융합 | GPU 와 동일 백본을 NPU 에서(분담/대조) |
| `cpu_embed_fusion` | CPU MobileNetV3 | CPU 융합 | 가속기 없는 PC 폴백 성능 |
| `gpu_embed_only` | GPU MobileNetV3 | 없음(코사인만) | 최속·정확도↓(융합 필요성 입증) |
| `gpu_npu_split_fusion` | **GPU+NPU 분담** | CPU 융합 | **임베딩을 두 장치에 절반씩** → 처리량↑ |
| `gpu_fusion_topk20` | GPU MobileNetV3 b16 | CPU 융합(top20) | 재채점 얕게 → 빠름(정확도 검증 필수) |
| `gpu_fusion_topk60` | GPU MobileNetV3 b16 | CPU 융합(top60) | 재채점 깊게 → 정확도 여유 |
| `gpu_fusion_crop` | GPU MobileNetV3 b16 | CPU 융합+중앙30% | 교차 호기 정확도↑ |
| `gpu_fusion_b1` | GPU b1 | CPU 융합 | **함정 재현**(처리량 폭락) |
| `gpu_fusion_b4` | GPU b4 | CPU 융합 | 처리량 정상화 지점 |
| `gpu_npu_ensemble_fusion` | GPU+NPU **앙상블(중복)** | CPU 융합 | **안티패턴 대조**(시간 2배·이득 0) |

> 정확도는 임베딩 백본 종류가 아니라 **CPU 고전 융합**이 좌우한다
> (docs/NPU 효율성 분석 보고서). 그래서 장치 조합은 주로 *"임베딩을 누가/어떻게
> 더 빨리 뽑느냐"* 의 속도 문제이며, 융합이 정확도를 지킨다.

---

## 4. 정확도 측정 방법

- **정답 라벨이 있으면**: `recall@1`, `recall@5`(상위 K 안에 정답이 있으면 정답).
- **라벨이 없으면**: `cpu_classical_full`(고전 전수, 정답 기준선) 의 top-1 과의
  **일치율** — '현행 정확도를 보존하는가'를 측정.
- **추천 규칙**: 완료된 조합 중 **현행(`gpu_fusion_b16`) 대비 정확도가 낮지
  않으면서 가장 빠른** 것. 정확도가 떨어지면 더 빨라도 추천하지 않는다.

---

## 5. 3배 속도 — 권장 탐색 순서

현행(`gpu_fusion_b16`)을 분모로, 정확도를 지키며 시간을 줄이는 경로:

1. **배치 고정** — GPU 는 `batch=16`(또는 4~16). `batch=1` 은 처리량 폭락(함정).
2. **재채점 깊이 조정** — `topk20` 이 정확도를 유지하면 CPU 단계가 짧아져 큰 이득.
   유지 못 하면 `topk40` 으로 되돌린다.
3. **임베딩 분담** — `gpu_npu_split_fusion` 으로 GPU+NPU 가 임베딩을 절반씩 동시에
   뽑아 추출 처리량을 올린다(중복 앙상블 아님).
4. **중앙 crop** — 교차 호기에서 정확도 여유가 필요하면 `gpu_fusion_crop`.

가장 빠른 조합이 **정확도 기준선과 동률**일 때만 채택한다.

> **검증 메모**: 임베딩 가속(GPU/NPU)의 실제 배속은 **Intel GPU/NPU 가 있는
> 대상 장비에서** 측정된다. 가속기가 없는 환경에서는 임베딩 레시피가 자동으로
> **CPU 고전 폴백**(비고에 표시)되어, 하니스 동작/정확도/기록은 그대로 검증되되
> 장치 배속만 0 으로 보고된다. 하니스는 장치-비의존이라 대상 장비에서 그대로
> 실측하면 된다.

---

## 6. 로딩 안전장치

- 레시피는 **순차** 실행(두 장치-부하 조합이 동시에 돌지 않음).
- 동시 추론 수는 **가용 메모리의 약 50% 안**에서 자동 상한(`safe_concurrency`).
- 조합별 **타임아웃**(초). 초과 시 그 조합만 중단하고 다음으로.
- **서브샘플**(slot/이미지 상한)로 대용량을 안전하게 표본 측정.
- 실행 중 **피크 메모리(MB)** 를 기록해 위험 조합을 사후 식별.
