# 매칭 속도 개선 — 진행 상황(개발자 모드 실험)

> 이 문서는 **현재 진행 중인 작업의 살아있는 기록**입니다. 무엇을 어떤 목적으로
> 실험하고, 무엇을 추구하며, 결과로 무엇을 할지를 한곳에 정리합니다.
> (최종 사용 설명은 `docs/개발자_벤치마크_가이드.md`, NPU 근거는
> `docs/NPU_효율성_분석_보고서.docx` 참고.)

- 작성/갱신: 2026-05-30
- 대상 브랜치: `claude/matching-npu-gpu-modes-GwTRB`
- 핵심 목표: **매칭(Stage 2) 속도 현행 대비 3배 — 단, 실제 매칭 정확도는 떨어지지 않게.**

---

## 1. 목표(무엇을 추구하는가)

1. **속도 3배**: 현행 `gpu_fusion_b16`(운영) 대비 매칭 단계 총시간을 1/3로.
2. **정확도 우선(절대 조건)**: 유사도 점수가 아니라 **실제 매칭 정답**(recall@1/@5,
   라벨 없으면 CPU 고전 전수 대비 top-1 일치율) 기준으로, 운영보다 낮아지면 **추천하지 않음**.
3. **로딩 안전**: 레시피 순차 실행 · 동시성 메모리 상한 · 레시피별 타임아웃 · 서브샘플.
   시스템이 로딩으로 꺼지지 않게.
4. **불필요한 실험은 하지 않음**: 함정/대조·폴백 중복·과거 저성능은 자동 스킵(아래 4절).

## 2. 무엇을 어떤 목적으로 실험하는가(레시피 그룹)

한 레시피 = **후보 선별(recall: 임베딩 추출, 장치 선택)** + **정밀 계산(scoring:
CPU 고전 pHash+ORB+SSIM 융합)**. "NPU로 데이터 뽑고 CPU로 계산" = `recall=NPU, scoring=fusion`.

| 그룹 | 키워드 | 목적(무엇을 알고 싶은가) |
|---|---|---|
| `core` (13) | 현행·기준선·사용자 아이디어 | 운영/정답 기준선 대비 어디가 빠르고 정확한지 큰 그림 |
| `npu-sweep` (29) | NPU 사용 방식 | **NPU를 어떻게 써야 가장 빠른가** — 모델·배치·동시추론(병렬)·성능힌트·스트림·전처리스레드·해상도를 한 축씩 |
| `npu-only` (5) | NPU 단독 | **CPU 재채점 없이 NPU 임베딩만으로 충분히 정확/빠른가** → 맞다면 CPU 불필요 |
| `fast-rerank` (6) | CPU 재채점 고속화 | **병목인 CPU 재채점을 줄이는 법** — ORB 생략/pHash 단독/병렬/topk 축소 |
| `model-zoo` (8) | 새 모델 | SuperPoint+LightGlue·PatchCore·PaDiM·CAE·U-Net·MobileViT 가 이 도메인에서 이득이 있는가 |

> 측정은 항상 **유사도 캐시를 우회**(`bench_no_cache`)해 '처음 매칭처럼' 공정 비교.

## 3. 지금까지의 핵심 발견(업로드된 실측 기반)

실측 `bench결과/`(809장·5,596쌍, GPU+NPU 감지, 정답 라벨 recall@1):

- **추천**: `npu_mbnet_cpu_fuse`(NPU 추출+CPU 융합) — 정확도 97.6% 유지하며 최속급.
  → 사용자 아이디어("NPU로 뽑고 CPU로 계산")가 유효함을 데이터로 확인.
- **병목은 CPU 재채점**: 임베딩 ~3.7s vs 재채점 ~57s. → 임베딩 장치(GPU/NPU)를 바꿔도
  총시간은 거의 그대로(현행 대비 **×1.02**, `npu_mbnet_cpu_fuse`). **속도 3배의 열쇠는
  '재채점 줄이기'**(ORB 제거·병렬화)이며 그래서 `fast-rerank` 가 핵심(`npu-sweep` 은 보조).
- **정확도를 깨는 지름길은 버린다**: `gpu_embed_only`(61%)·`gpu_npu_split_fusion`(80%)·
  `gpu_fusion_topk20`(90%)·`gpu_fusion_crop`(93%)은 운영(97.6%)보다 낮아 **자동 스킵 대상**.
- 자기검증(CPU, 가속기 없는 환경)에서 `fast-rerank`의 `rr_phash`가 전수 대비 **~6배**,
  `rr_parallel`이 **~3배** 빠르며 정확도 100% 유지(데이터 의존 — 실데이터 재확인 필요).

## 4. 불필요한 실험을 건너뛰는 규칙(자동 스킵)

기본 스위트는 다음을 **측정하지 않고** 사유와 함께 '건너뜀'으로 기록한다(명시 선택 시 예외):

1. **함정/대조용**(`diagnostic`): `gpu_fusion_b1`(batch1 함정, 과거 891초)·
   `gpu_embed_only`·`gpu_npu_ensemble_fusion`. → 운영 판단에 새 정보 없음.
2. **폴백 중복**: 필요한 GPU/NPU가 없거나 모델 패키지(timm/kornia/anomalib)가 없어
   어차피 CPU 고전으로 폴백 → 같은 결과 반복이라 스킵.
3. **과거 저성능**: 이전 실험 기록에서 **운영 대비 정확도가 3%p 넘게 낮았던** 레시피.
   (가속기 없는 CPU 폴백 기록은 변별력이 없어 판정에서 제외. 근소차 ≤3%p는 노이즈로 관용.)

해제: CLI `--all-recipes` 또는 GUI '모든 레시피 측정' 체크.

## 5. 결과로 무엇을 할 것인가(다음 행동)

- **즉시 채택 후보**: NPU 추출+CPU 융합 계열이 운영 정확도를 지키며 가장 빠르면,
  운영 기본값을 그쪽으로 옮기는 것을 검토(정확도 동률 확인 후).
- **속도 3배 경로 탐색 순서**:
  1. `npu-sweep`로 **NPU 추출 최적 노브**(배치·동시추론·스트림·전처리스레드) 확정.
  2. `fast-rerank`로 **CPU 재채점 시간**을 정확도 손실 없이 최대한 절감(ORB 비용이 큼).
  3. 1+2 결합(`rr_npu_phash_parallel`: NPU 추출 + ORB 뺀 병렬 재채점)으로 3배 도전.
  4. `npu-only`가 정확도를 만족하면 CPU 재채점 자체를 제거(가장 큰 속도 이득).
- **추천 규칙은 불변**: 운영 정확도 미만이면 더 빨라도 채택하지 않는다.
- **검증 방법**: 정답 라벨(`정답 라벨 만들기`)로 recall@K 측정. 장치 배속은 Intel
  GPU/NPU 대상 장비에서 실측(가속기 없는 환경은 CPU 폴백으로 하니스만 검증).

## 7. 실측 결론(2026-05-31) + 중앙-인식(center-aware) 방향 + NPU 재정의

### 7.1 실측으로 확정된 것 (4개 런)
- **병목 = CPU 재채점**(임베딩 ~4s vs 재채점 ~57s). 임베딩 장치 교체는 ×1.02 — **데드엔드**.
- **재채점 병렬화가 3배 레버**: `rr_parallel` = **97.6%(현행 동률) @ ~17s** = 현행(62~75s) 대비 **×3.6~4.3**.
- **ORB 가 정확도를 지탱**: ORB 유지=97.6% 전부, ORB 제거=41~71% 붕괴. → "ORB 제거로 가속"은 폐기,
  "**ORB 병렬화/특징수 축소**"가 정답.
- **NPU 임베딩은 batch=1 만 정확**(b4/b8/b16/동시추론은 78~88%로 붕괴). 느린 데다 부정확.
- 신뢰성 경고: **GT 41쿼리(1장=2.44%p)** — 97.6% 동률은 정확도로 구분 불가. 단일 측정·런간 분산 큼.
  추천 엔진이 production 미포함 런에서 `cpu_classical_full`(228s)을 잘못 추천 → **린 기본에 현행 항상 포함**으로 수정.

### 7.2 기준 이미지 분석 — defect 은 정중앙 (사용자 통찰)
- `기준/` 51장(6웨이퍼)·전부 1000×1000·defect 3종(Irregular Bump/Foreign Material/Repassivation).
- 파일명 좌표로 **AOI 장비가 defect 좌표에 FOV 를 맞춰 크롭** → **defect = 중앙의 작은(~30~80px) 신호**,
  주변은 반복되는 RDL 패턴(via·트레이스·패드).
- 함의: 전역 디스크립터(임베딩·pHash·풀이미지 ORB)는 **배경(반복 패턴)이 지배** → 혼동원. 중앙 defect 은
  작아 과소평가. 그래서 `gpu_fusion_crop`(중앙30%만)은 **오히려 정확도↓(92.7%)** — 맥락(주변)을 잃어서.
  → **풀이미지도 중앙만도 아닌, 둘을 단계/가중으로 쓰는** 접근이 맞다.

### 7.3 신규 center-aware 레시피 (`--recipes center`)
- **A. 영역분해 융합**(`center_fusion_*`): 중앙(defect) crop 유사도 + 풀 ROI(주변 패턴) 유사도를
  따로 계산해 **중앙 가중**으로 융합(한 패스). 비율 r·가중 w 스윕(r0.20~0.30, w0.5~0.7) + 병렬 변형.
- **B. 거친→정밀 캐스케이드**(`center_cascade_*`): ①중앙 점수로 topk 를 keep 개로 거칠게 추리고
  ②비싼 풀 ROI 고전을 그 keep 개에만 → 풀 재채점 횟수↓로 **속도↑**. keep·병렬 변형.
- 배선: `benchmark.blend_region_scores`/`cascade_survivors`(순수·테스트됨) + `_rerank_score_map`(채점 경로).
  중앙 비율은 `SimilarityConfig.center_ratio`(0=레거시 0.3) → `image_io.center_roi_gray` 가 적용.

### 7.4 개발자 테스트 목록 정리 (린 기본 + 아카이브)
- **린 기본(`quick`)**: 정확도 gold·현행 + 재채점 생존자(`rr_parallel`/`cpu_rr_orb_only`/`cpu_rr_parallel16`)
  + center-aware 4종. **현행을 항상 포함**해 추천이 항상 유효.
- **대결(`faceoff`)**: 현행 vs 재채점 생존자 직접 비교(반복 측정으로 분산 확인용).
- **아카이브(opt-in)**: 입증된 사패(임베딩 장치 교체·ORB 제거 붕괴·NPU 배치)는 그룹/`all+` 로만.

### 7.5 NPU 재정의 + 배치 정확도 버그 진단 (먼저 할 일)
- NPU 임베딩(풀이미지)은 **속도이득 0 + 배치 정확도 손상** → 임베딩 가속 노선 종료.
- 의미 있는 NPU 역할은 **CPU 재채점과 겹쳐 도는 별도 작업**(예: 중앙 defect 인코딩, batch=1)뿐 →
  단, **배치 정확도 붕괴 원인 규명이 선행**.
- 진단 도구: `benchmark.diagnose_npu_embedding` + CLI **`--npu-embed-diag --ref 기준`** —
  같은 이미지에서 **GPU vs NPU(b1) vs NPU(b16)** 임베딩의 코사인/L2 차이를 측정.
  `npu_b1|npu_b16` 코사인이 1.0 에서 멀수록 "배치가 임베딩을 바꿔 후보 선정을 망친다"는 직접 증거.

5. 승자 운영 경로 승격 + recall ≥ 현행 회귀 테스트.

## 9. 생존자 정리 + 창의적 신규(중앙-가중 ORB) + 운영 병렬 확정 (2026-05-31)

### 9.1 옵션 정리 — '생존자'만 노출
- 개발자 모드 옵션을 **앵커(gold·현행) + 생존자(97.6% 보존 확인)** 와 신규 실험만 보이게 정리.
  입증된 사패(임베딩 장치 교체·ORB 제거 붕괴·NPU 배치·center-aware 비효율·모델주머니)는
  옵션에서 내리고 아카이브(`--recipes all+`/그룹)로만 둔다.  키: `SURVIVOR_KEYS`/`MAIN_KEYS`.

### 9.2 ★ 발견 — 운영은 이미 재채점을 병렬화한다
- `efficiency_matcher.EfficiencyScheduler._consume_slot` 이 ref별 재채점을 **코어 수(≤32)
  스레드풀로 병렬** 실행하고, val 특징을 슬롯당 1회만 공유 추출하며, GPU 임베딩 ∥ CPU
  재채점을 파이프라인한다.  즉 **운영 속도는 벤치의 직렬 `gpu_fusion_b16`(62s)이 아니라
  `rr_parallel`(~15s)에 가깝다** — 3배 레버(병렬)는 이미 운영에 들어가 있다.
- '확정·고착': 직렬 회귀를 막는 회귀 테스트 추가(`_rerank_workers()>=2` + 스케줄러가 풀에
  `_rerank_one` submit 유지).  벤치 직렬 anchor 는 '병렬의 가치' 대조로만 본다.

### 9.3 창의적 신규 — 중앙-가중 ORB(단일 패스)
- 영역분해(A)·캐스케이드(B) 실패 원인은 **파이프라인이 ROI 를 384px 로 재정규화**해
  중앙 crop 이 비용을 못 줄이고 2패스가 된 것.  교정: **추가 패스 없이 ORB 매치를
  '중앙(defect) 근접도'로 가중**(`orb_center_weight`).  배경(반복 패턴) 매치 영향을 줄여
  defect 판별력↑ — 비용은 현행과 거의 동일.
- 배선: `orb.compute_orb` 가 키포인트 좌표 보관 → `Feature.orb_xy`(캐시 호환·구캐시는 None)
  → `pipeline.score(center_strength=)` → `orb.orb_score`.  순수 가중 함수
  `centrality_weights`/`centrality_weighted_ratio`(strength=0 이면 현행과 동일) 헤드리스 테스트.
- 신규 레시피(`--recipes orb-center`, 옵션 노출): `rr_orb_center50/70`(ORB 단독+중앙가중·병렬),
  `rr_fusion_center50`(전체 융합에서 ORB 항만 중앙가중).  대상 장비에서 recall 측정 예정.

- 측정: `orb-center` 가 현행 97.6% 를 **넘는지/보존하는지** 대상 Intel 장비에서 확인.
